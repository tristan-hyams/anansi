## Work Done

### Phase 5 — weaver package

Originally named `crawler`, renamed through `nest` to final `weaver` — thematic naming after Anansi the spider. The weaver weaves the web, crawlers venture out to fetch pages.

- **Weaver** — orchestrator. Owns config, frontier, rate limiter, robots rules. Pre-creates Crawlers during `NewWeaver()`.
- **Crawler** — worker goroutine. Fetches pages, checks Content-Type and X-Robots-Tag, parses links, normalizes/filters, enqueues to frontier.
- **Web** — crawl result struct with `String()` method for formatted summary. Sorted by depth then alphabetically.
- **WeaverConfig** — decoupled from CLI config. `Validate()` and `CrawlRate(rules)` methods.
- **81.9% coverage**, 8 httptest integration tests + 1 live crawlme.monzo.com test.

### Phase 6 — CLI wiring

- `main.go` wires `ParseFlags → OriginURL → NewWeaver → Weave → web.String()`
- Exit codes: 0 success, 1 error, 130 SIGINT
- Structured JSON logs to stderr, summary report to stdout
- `make run` crawls the default target end-to-end

### webutil package

- **Singleton `Transport()`** via `sync.Once` — shared connection pool across all HTTP clients
- **`NewClient(timeout)`** — per-worker client wrapping the singleton transport
- Tuned for single-domain crawling: 100 idle conns per host, HTTP/2, 10s dial/TLS/response-header timeouts

### Naming Changes

- `crawler` package → `weaver` (via `nest`) — avoids `crawler.Crawler` stutter
- `Result` → `Web` — the web that was woven
- `seed` → `origin` everywhere — `AnansiConfig.Origin`, `OriginURL()`, `normalizer.IsSameHost(origin, candidate)`
- `Run()` → `Weave()` — thematic
- `worker()` → `Crawler.crawl()` — the thing that crawls

### robots.Fetch simplification

Moved HTTP client creation inside `robots.Fetch()` — it creates its own client via `webutil.NewClient(fetchTimeout)`. Callers no longer manage client creation for a one-shot setup call.

## Design Decisions

### Pre-created Crawlers
Crawlers are created once in `NewWeaver()` and stored in a `[]*Crawler` slice. Each gets its own `http.Client` backed by the shared transport singleton. `Weave()` launches them without allocation.

### Active work tracking race fix
The monitor checks **both** `active == 0 AND queue.Len() == 0` to detect natural completion. This avoids a race where a crawler is between `Dequeue` returning and `active.Add(1)` — the queue would be empty and active would be 0 briefly, but the URL is about to be processed.

### Web.String() on the result type
Summary formatting moved from CLI to `Web.String()` method — any consumer of the weaver package gets formatted output, not just the CLI. Implements `fmt.Stringer`.

### CrawlRate as Config method
Rate limit calculation (config rate vs robots.txt Crawl-delay) extracted to `WeaverConfig.CrawlRate(rules)` — testable, single responsibility.
