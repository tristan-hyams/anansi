**Date:** 2026-04-15

## Context

Continuing from session 03 (normalizer + parser). This session built the frontier package — the URL queue and dedup layer.

## Work Done

### Phase 3 — frontier

- **frontier.go** — `Frontier` interface with `Enqueue`, `Dequeue`, `Clear`. `InMemory` implementation using buffered channel + `sync.Map`.
- **FrontierURL struct** — wraps `*url.URL` with `Depth`, `Status`, and `Err`. No parent pointer — a page has many links (1:many), so a single parent reference is misleading and creates a one-way chain that can't be traversed downward. Depth is set by the caller.
- **Status enum** — `StatusPending`, `StatusVisited`, `StatusError`.
- **Dedup in Enqueue** — `sync.Map.LoadOrStore` checks and stores in one atomic operation. Duplicates logged at debug level.
- **Buffer size default** — if caller passes < 1, defaults to 100,000 (in `consts.go`). The visited set is the real bound on growth, not the channel.
- **Clear()** — drains channel and resets visited set.
- **100% coverage**, 10 tests.

### Design Decisions

#### WaitGroup removed from frontier
Originally the frontier owned a `sync.WaitGroup` for completion detection. Removed — lifecycle management belongs in the crawler. The frontier is a pure queue + dedup layer. The crawler manages its own worker WaitGroup and cancels the context when done, which unblocks any pending Enqueue/Dequeue via `ctx.Done()`.

#### Parent pointer removed from FrontierURL
Initially considered `Parent *FrontierURL` to form an implicit tree for path reconstruction. Rejected — the relationship is 1:many (a page links to many URLs), so a single parent reference only captures one discovery path. Without `Children []` linked back, it's a one-way chain. Depth is a simple int set at enqueue time instead.

#### select {} behavior investigation
Deep investigation into Go's `select` multiplexing. Historical Go versions had issues where `select` could commit to evaluating one blocking case and miss another becoming ready. Wrote raw channel tests (no frontier code) proving modern Go multiplexes correctly:
- `TestSelect_ContextCancelUnblocksBlockedChannel` — ctx.Done() fires while channel send is blocking
- `TestSelect_ChannelUnblocksWhileContextPending` — channel send fires while ctx.Done() is pending

Both prove the `select` picks up whichever case fires first without a `default` + sleep loop.

#### Buffer size philosophy
Discussed why Go's culture demands bounded channels while C#'s `Channel.CreateUnbounded<T>()` is the default. For our use case, the visited set prevents unbounded growth — the channel buffer just needs to be large enough to never block writers. 100k default is generous for single-subdomain crawls.

### What's missing (documented, not yet implemented)
- **Queue metrics** — `Len()`, `VisitedCount()` for summary report
- **Retry support** — re-enqueue failed URLs (bypassing dedup)
- **Priority** — FIFO is fine for now, but production crawlers use priority queues
- **Max depth enforcement** — frontier has the data (`FrontierURL.Depth`), but enforcement lives in the crawler
